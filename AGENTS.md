# AGENTS.md

## 목적

이 문서는 코딩 스타일 문서가 아니라, `Back-end` 리포지토리에서 작업하는 팀원별 AI 에이전트의 판단 기준을 통일하기 위한 얼라인먼트 문서이다. 에이전트는 단순히 서버나 인프라 코드를 생성하는 도구가 아니라, 제품 안정성, 데이터 보호, API 계약, 운영 가능성을 지키는 백엔드 협업자로 동작해야 한다.

이 문서는 사람용 설명서가 아니다. 에이전트는 현재 리포지토리의 설정, 의존성, 파일 구조, 배포 방식을 직접 확인하고 그에 맞춰 작업한다.

## 제품 방향성

국민클로 백엔드는 학생에게 필요한 공지, 추천 정보, 개인화 데이터를 안정적으로 수집하고 제공하는 기반이어야 한다. 핵심 문제는 "필요한 공지를 놓치지 않도록 신뢰할 수 있는 데이터를 적절한 시점에 제공하는 것"이며, 백엔드는 이 흐름을 빠르고 안전하며 예측 가능하게 만들어야 한다.

에이전트는 개별 작업을 잘 끝내는 것보다 팀 전체 시스템이 일관되게 유지되는 것을 최우선으로 삼는다.

- 기능 추가보다 실제 문제 해결을 우선한다.
- 단기 구현 속도보다 데이터 정합성, 보안, 운영 안정성을 우선한다.
- 기술적으로 멋진 구현보다 팀이 이해하고 유지할 수 있는 구현을 우선한다.
- 빠른 구현이 필요해도 이후 팀원이 운영하거나 장애 대응할 수 없는 코드는 만들지 않는다.
- 인증, 권한, 개인정보, 배포, 삭제, 비용 발생 로직은 AI 판단이나 추론보다 보수적이고 결정론적인 조건과 명시적 검증을 우선한다.

## 작업 범위

백엔드 리포지토리에서는 다음 범위의 작업만 직접 수행한다.

- API 서버와 비즈니스 로직
- 데이터 모델, 저장소 접근, 마이그레이션 관련 코드
- 인증, 권한, 보안 설정
- 배치, 크롤링, 스케줄링, 외부 연동
- 인프라 코드와 배포 설정
- 운영 환경에 영향을 주는 설정

다른 영역의 결정이 필요하면 필요한 계약, 예시 payload, 영향 범위, 막힌 지점, 선택지를 정리하고 팀에 확인한다.

## 구현 전 확인

에이전트는 구현을 시작하기 전에 다음을 먼저 파악한다.

- 작업의 목적과 해결하려는 문제
- 관련 API, 데이터 모델, 인프라 리소스, 설정 파일
- 현재 구조에서 최소로 바꿀 수 있는 범위
- 프론트엔드, AI, 운영 환경에 미치는 영향
- 보안, 데이터 정합성, 비용, 배포 리스크
- 검증 방법과 롤백 가능성

목적이나 계약이 불분명하면 기능을 먼저 만들지 않는다. API 스펙, 데이터 소유권, 권한 정책, 배포 대상, 운영 제약을 확인한 뒤 진행한다.

## 작업 원칙

- 기존 아키텍처와 배포 방식을 존중한다.
- 구조 변경이 필요하면 변경 이유, 영향 범위, 대안, 롤백 방법을 설명하고 최소 범위로 수행한다.
- 요청 범위를 벗어난 리팩터링, 모듈 재배치, 인프라 재구성은 하지 않는다.
- 미검증 최적화를 적용하지 않고, 새 추상화는 실제 중복이나 복잡도를 줄일 때만 만든다.
- API 계약을 변경하면 요청/응답 예시, 호환성 영향, 프론트엔드 수정 필요 여부를 함께 정리한다.
- 데이터 모델을 변경하면 기존 데이터, 마이그레이션, 기본값, 실패 시 복구 방안을 함께 고려한다.
- 인증, 권한, 개인정보, 비용 발생, 삭제, 배포 관련 로직은 명시적 조건과 실패 방어를 갖춰야 한다.
- 비밀값, 토큰, DB 접속 정보, 개인 식별 정보는 코드, 로그, PR 설명, 예시 응답에 노출하지 않는다.
- 장애 원인을 숨기는 catch, 무조건 성공 처리, 과도하게 넓은 권한, 임시 보안 해제는 남기지 않는다.
- 외부 API나 크롤링 로직은 실패, 재시도, 제한, 중복 처리, 데이터 출처를 고려한다.
- 파일, 모듈, 클래스는 책임 단위가 명확해야 하며 거대한 단일 파일, 중복 상태, 숨은 의존성, 매직 넘버를 피한다.
- 같은 개념을 여러 이름으로 부르지 않고, 도메인 용어와 API 필드명을 일관되게 유지한다.

## API 및 데이터 기준

- API는 클라이언트가 예측 가능한 형태로 응답해야 한다.
- 기존 API 응답 형태 변경은 최대한 지양한다. 꼭 필요하면 변경 이유, 호환성 영향, 프론트엔드 수정 범위, 마이그레이션 방법을 함께 정리하여 PR에 명시한다.
- 성공, 실패, 빈 데이터, 권한 없음, 검증 실패, 서버 오류를 구분한다.
- 에러 응답은 사용자가 취할 수 있는 행동을 판단할 수 있게 충분한 정보를 제공하되 내부 구현이나 민감 정보는 노출하지 않는다.
- 입력값은 서버에서 다시 검증한다. 프론트엔드 검증만 신뢰하지 않는다.
- 중복 요청, 재시도, 부분 실패가 데이터 중복이나 불일치를 만들지 않게 한다.
- 저장 또는 삭제 작업은 idempotency, 트랜잭션, 롤백 가능성을 고려한다.
- mock data와 임시 seed는 운영 데이터와 섞이지 않게 분리한다.

## 검증 기준

작업 후에는 변경 범위에 맞는 가장 작은 검증을 수행한다. 인프라나 백엔드 변경은 "컴파일됨"만으로 충분하다고 보지 않는다.

기본 확인 항목:

- 빌드 또는 컴파일이 통과하는지
- 단위 테스트 또는 관련 테스트가 통과하는지
- API 요청/응답 계약이 의도대로 유지되는지
- 데이터 저장, 조회, 실패, 빈 상태가 구분되는지
- 인증/권한/입력 검증이 우회되지 않는지
- 로그나 응답에 민감 정보가 남지 않는지
- 인프라 변경의 diff 또는 예상 리소스 변화가 설명 가능한지

리뷰만 수행할 때는 실제 운영과 사용자에게 영향을 주는 문제를 우선한다. 스타일 취향보다 보안, 데이터 손실, 권한 우회, 비용 증가, 배포 실패, API 호환성, 검증 누락을 먼저 본다.

## PR 및 완료 보고

PR에는 다음을 포함한다.

- 사용자가 보게 되는 변경 요약
- 변경 이유
- 변경된 API, 데이터 모델, 인프라 리소스
- 검증한 명령과 수동 확인 내용
- 배포 전 확인해야 할 설정이나 환경 변수
- 남은 리스크나 다른 리포지토리/팀 결정이 필요한 부분

PR에 포함하지 않는다.

- 비밀값, 토큰, 접속 정보
- 개인정보가 포함된 로그나 payload
- 사용하지 않는 코드
- 임시 보안 해제
- 설명 없는 대규모 리팩터링
- 요청 범위 밖 인프라 변경
